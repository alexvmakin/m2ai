# Ревизия процедуры разбора ГП (v1) и план прогона v2

> Дата: 2026-06-09 · Вход: SKILL.md, METHODOLOGY_OF_WORK.md, GP_DECOMPOSITION_ALGORITHM_AND_EXPERIMENT.md, GP_OPERATIONAL_LAYER_ONTOLOGY.md, GP_x_LLM_RESEARCH_PLAN.md, ENTITIES/ (333 сущности), навигаторы, 5 JSON-разложений.

## 1. Что было сделано (две независимые линии)

Проект де-факто шёл двумя слабо связанными линиями:

**Линия А — Извлечение сущностей (entity-base).** Методика `METHODOLOGY_OF_WORK.md`: чтение книг по 20 стр., извлечение в 6 категорий (concepts/schemas/theses/methods/methodologies/theories), на каждую сущность — `entity.md` + `LOG.md` + `CONNECTIONS.md`, дельта-анализ (🆕/🔄/✅/⚠️), мастер-индекс. Результат: 333 сущности из Папки 0 + Книг 1–3, HTML-навигаторы.

**Линия Б — GP-разложение (operation-base).** Скилл `gp-decomposition`: нисходящее функционально-генетическое разложение текста на операции (сопоставление S1–S6 + отнесение object/value/law), алфавит типов, генетическая карта (DAG). Результат: 3 разложения + сводный алфавит (18 типов) + валидация PASSED, пилот на Gettier (8 типов операций, цикл DEFINITION→VALIDATION→REFUTATION→REPAIR).

Поверх обеих линий уже спроектирован, но не реализован, **операционный слой** (`GP_OPERATIONAL_LAYER_ONTOLOGY.md`): архитектура Palantir-style Hydrate→Ontology→Wield, типизированный граф объектов, entity resolution как ключевая нерешённая задача.

## 2. Сильные стороны

- **Двухчастный инвариант** (сопоставление+отнесение) — даёт детерминированную «клетку» для стохастической LLM; воспроизводимо.
- **Дельта-протокол** извлечения (🆕/🔄/✅/⚠️) — корректно отслеживает приращение знания по источникам.
- **Валидация на контрольном фрагменте** (0 новых типов на §§58–60) — методологически честная проверка замыкания алфавита.
- **Генетическая карта** — не плоский список типов, а порядок «проще→сложнее», это и есть содержательно-генетическая логика в действии.

## 3. Слабые места (что чинить в v2)

1. **Две линии не сшиты.** Сущности (entity-base) и операции (operation-base) живут отдельно. Операция «определение JTB» нигде не связана с сущностью-понятием, которое она производит. Нет общего графа объектов.
2. **Отдельные артефакты на статичных HTML.** Навигаторы — это снапшоты-файлы, а не живая база. Нельзя серфить, нет сквозных переходов entity↔operation↔source↦цитата.
3. **Нет единой адресации и entity resolution.** ID разнятся (C###/S###/OP-###/A##), один концепт под разными slug'ами в разных книгах не схлопывается. Это прямо назван «ключевой нерешённой задачей» в операционной онтологии.
4. **Слабая привязка к первоисточнику.** В entity.md есть «стр. 236», но нет точной цитаты-якоря и ссылки на фрагмент PDF; нельзя кликнуть с операции на конкретное место в книге.
5. **Разложение не масштабировано.** Сделано на 3 текстах + 1 пилот. Нет пайплайна для массового прогона и контроля качества на десятках/сотнях текстов.
6. **Отчёты неинтерактивны и неглубоки.** JSON + HTML-дашборд — хорошо для демо, плохо для исследовательской работы: нет drill-down, фильтров по S-типу, сравнения текстов, экспорта подграфов.
7. **Нет хронологии.** Тексты ГП 1956→1987 не уложены в таймлайн развития его мышления (хотя триада S2+S5+S6 как инвариант 30 лет — уже намечена).

## 4. Целевая модель v2: единый граф связанных объектов

Свести обе линии в один типизированный граф (Hydrate→Ontology→Wield), где узлы:

`Source` (книга/глава/страница) · `Fragment` (цитата-якорь) · `Operation` (S-тип + отнесение) · `OperationType` (A01–A18) · `Entity` (концепт/схема/тезис/метод/методология/теория) · `Author` · `Date`.

Рёбра: `produces` (operation→entity), `appears_in` (entity→fragment), `cites` (fragment→source), `transforms` (type→type, генетика), `precedes` (date→date, таймлайн), `refines` (entity→entity, дельта-версии), `same_as` (entity resolution).

Над этим графом — **вики-навигатор**: каждая статья (Entity/Operation/Source) — страница со сквозными ссылками, обратными ссылками (backlinks), цитатами-якорями и мини-графом окружения. База — связанные объекты; интерфейс — серфинг по статьям.

## 5. План прогона v2 (улучшенная процедура)

**Шаг 0. Унификация адресации.** Единая схема ID и резолвер сущностей (`same_as`) — схлопнуть дубликаты по книгам. Привязать каждую сущность к ≥1 `Fragment` с точной цитатой и страницей.

**Шаг 1. Re-ingest в граф.** Перелить 333 сущности + 5 разложений в типизированный граф (JSON/graph-формат). Сшить operation `produces` entity.

**Шаг 2. Углублённое разложение с богатым протоколом.** К текущим полям операции добавить: цитату-якорь, латентность эталона, «конвенциональный эталон» (новый тип из пилота), уверенность, ссылки на производимые сущности.

**Шаг 3. Глубокие интерактивные отчёты.** Не статичный HTML, а вики-навигатор + дашборды с drill-down: фильтр по S-типу/уровню знака/тексту, сравнение алфавитов между текстами, экспорт подграфа.

**Шаг 4. Хронология.** Уложить тексты на ось 1956→1987; показать, как менялся набор операций и какие инварианты держатся.

**Шаг 5. Контроль качества.** Прогонять валидацию на контрольных фрагментах каждого нового текста; метрика «доля операций, покрытых существующим алфавитом» как индикатор замыкания.

Этот план — вход для проектов **P2 (вики-база)** и **P4 (массовое разложение и таймлайн)** дорожной карты.
