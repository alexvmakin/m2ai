# P8 — Workbench + RAG v2 · спецификация

> Цель: превратить m2ai из «вики + поиск» в полноценную рабочую среду общения с базой ГП —
> быстрые вопросы, несколько чатов, создание и сохранение статей, работа с артефактами,
> плюс «правильный» RAG (вектор + граф + онтология) и пайплайн пополнения базы с админкой.

## 1. Архитектура — 4 слоя

```
L4  Ingestion + Admin   пайплайн пополнения базы + простая админка (CRUD, ревью, реиндекс)
        │ upsert (граф + вектор), версии (revises), вердикт покрытия
        ▼
L1  Knowledge base of record   наш онтологический граф (nodes/edges, 7 слоёв, S-типы,
        │                       якоря, same_as) — git-версионируемый, канонический
        ▼
L2  RAG v2 retrieval    pgvector (эмбеддинги) + BM25 + обход графа (multi-hop) + rerank
        │ → связный подграф-контекст + цитаты + грунтинг
        ▼
L3  Interaction (Workbench)   Open WebUI: чаты, артефакты, статьи, рабочие пространства,
                              сохранение; модели через RMR; ретрив через наш GraphRAG (Pipelines)
                              + лёгкий публичный вход /ask (как сейчас)
```

Принцип: **граф остаётся источником правды у нас**, готовый UX (чаты/артефакты/админ) берём
из опенсорса и подключаем к нашему ретриву и к RMR.

## 2. Функциональные требования (сценарии)

**S1 · Быстрый вопрос.** Поле ввода → grounded-ответ по подграфу с цитатами `[ID]` и
источниками книга+страница; уточняющие вопросы в одном треде. (Уже есть в /ask, переносим в чат.)

**S2 · Создать и сохранить статью по теме / нескольким темам.** Пользователь выбирает 1–N
тем/сущностей (или задаёт тему текстом) → система собирает релевантный подграф → RMR пишет
структурированную статью (с разделами и цитатами) → сохранение как **артефакт/документ**
(править, версионировать, экспорт md/pdf), опционально — публикация обратно в базу как entity.

**S3 · Работа с артефактами.** Канвас артефактов: статья, диаграмма (мини-граф/таймлайн), таблица.
Редактирование, версии, экспорт, прикрепление к чату, повторное использование.

**S4 · Несколько чатов / рабочие пространства.** Постоянные треды, организованные по проектам/темам,
поиск по истории, переключение моделей (через RMR).

**S5 · Сохранение и повторное использование.** Сохранять чаты, статьи, артефакты; ссылаться на
сохранённые сущности; «закреплять» материалы в базе знаний (knowledge base).

**S6 · (Админ) Пополнение и обновление базы.** Загрузить новый источник → прогон пайплайна →
ревью (same_as, пробелы) → публикация; правка существующих сущностей; реиндекс.

Acceptance (MVP): S1 + S2 + S4 работают на проде; S3/S5 — базово (артефакт-статья сохраняется);
S6 — простая админка с CRUD + реиндекс.

## 3. Технологическое решение — какой опенсорс

| Кандидат | Сильные стороны | Роль у нас |
|----------|-----------------|-----------|
| **Open WebUI** (рекоменд.) | Полированный чат, гибридный RAG (BM25+вектор+rerank), knowledge bases, RBAC/админ, **Pipelines** (OpenAI-совместимый фреймворк плагинов → наш GraphRAG как ретривер), OpenAI-совместимые провайдеры (RMR), артефакты, MCP | Основной Workbench (L3) |
| LibreChat | Мульти-провайдер, агенты, ветвление диалога, RAG-API (Meilisearch гибрид), артефакты (HTML/React), MCP, корпоративная авторизация | Альтернатива L3, если важны агенты/ветвление |
| LobeChat | Лучший UX, артефакты, мульти-агент, маркетплейс MCP | Альтернатива L3 для «продуктового» вида |
| Dify | Конструктор RAG-приложений и workflow, knowledge-админка, ингест | Кандидат на L4 (админка/пайплайн) |

**Рекомендация:** взять **Open WebUI** как Workbench. Почему: (1) OpenAI-совместим → RMR
подключается напрямую как провайдер; (2) **Pipelines** — это OpenAI-совместимый плагин-слой, в
который мы кладём наш Ontological GraphRAG как кастомный ретривер (наш граф остаётся движком,
UI бесплатно); (3) встроенные knowledge bases + RBAC + админ + артефакты закрывают S2–S6; (4)
self-host, активная разработка, MCP. Ближайший конкурент — LibreChat (брать, если приоритет —
агенты и ветвление). Финальный выбор — за тобой (вопрос ниже).

## 4. RAG v2 — вектор + граф + онтология

Сейчас MVP: BM25 + RMR-эмбеддинги (in-memory) + 1-hop расширение. v2:

- **Vector store:** pgvector в Postgres (Railway-native, у вас уже есть managed Postgres в X-Ray)
  — эмбеддинги сущностей и чанков, персистентно (не пересобираем при каждом деплое).
- **Граф (онтология):** наш типизированный граф = «онтологический» слой. GraphRAG-обход:
  vector находит seed-узлы → traversal по рёбрам (relates_to/refines/produces/transforms) собирает
  **связный подграф-контекст**; для «глобальных» вопросов — community-summaries (по слоям/кластерам).
- **Гибрид:** vector (семантика) + BM25 (лексика) + graph-expansion + rerank (cross-encoder или
  LLM-rerank через RMR). Грунтинг цитат — как сейчас.
- **Фреймворк (опц.):** взять паттерны **LightRAG** (dual-level: локальный entity-retrieval +
  глобальный graph-retrieval; бэкенды Postgres/Neo4j) — но кормить его **нашим курируемым графом**,
  а не авто-извлечённым (наш точнее). Либо доращиваем свой движок (минимум новых зависимостей).
- **Оценка:** наш rag_eval (recall@k/MRR) + RAGAS (faithfulness/answer relevancy) на проде.

## 5. Пайплайн пополнения и обновления базы

Операционализация методики M2-Граф (A0→A6) как повторяемого пайплайна:

```
intake (книга/текст/транскрипт)
  → decompose (GP-операции) + extract (сущности)
  → resolve (same_as к существующему графу) + anchor (цитаты+страницы)
  → enrich (пересказ, IP-safe) + version (revises, дата)
  → embed (pgvector) + upsert (граф + вектор)
  → verdict (A4.5 сверка с каркасом / A4.99 покрытие: GO/CAUTION/LOOP)
  → publish (вики + RAG + экспорт)
```

Каждый шаг логируется; LOOP возвращает на дозаполнение. Версионность инстансов — уже в модели.

## 6. Админка v1 (простая, дорабатываем потом)

Scope MVP: список сущностей с фильтром по заполненности; добавить/править сущность (заголовок,
определение, слой, связи, якоря); **доска ревью same_as** (уже есть); кнопка «реиндекс» (пересборка
граф+эмбеддинги); загрузка нового источника; отчёт покрытия (уже есть). Доступ — за авторизацией.

Технически (на выбор): (а) тонкая своя админка в портале (CRUD над graph JSON → git-commit →
rebuild) — минимально и под контролем; (б) knowledge-админка Dify/Open WebUI для векторного слоя.
Рекоменд.: своя тонкая админка для графа + векторный слой через Open WebUI/pgvector.

## 7. Интеграция с текущим m2ai

m2ai-портал остаётся **публичной канонической вики + API** (/wiki, /api/graph, /api/rag, /api/documents).
Workbench (Open WebUI) — отдельный сервис, читает ретрив у нас (через Pipeline, бьющий в /api/rag v2)
и модели у RMR. Так не ломаем готовое и переиспользуем граф.

## 8. Фазы P8

- S0 · Поднять Open WebUI (Railway/docker), подключить RMR как провайдера, базовые чаты/артефакты.
- S1 · Pipeline-плагин: наш GraphRAG как ретривер в Open WebUI (S1/S2 сценарии).
- S2 · RAG v2 бэкенд: pgvector + graph-traversal + rerank (заменяет in-memory).
- S3 · Пайплайн пополнения + простая админка (S6).
- S4 · Артефакты/статьи (S2/S3/S5), экспорт, сохранение в базу.
- S5 · Оценка (RAGAS), community-summaries для глобальных вопросов.

## 9. Решения к фиксации

1. Опенсорс-Workbench: **Open WebUI** (реком.) vs LibreChat vs LobeChat.
2. Vector store: **pgvector** (реком.) vs Qdrant.
3. GraphRAG: доращиваем **свой движок** (реком., минимум зависимостей) vs LightRAG.
4. Хостинг Workbench: отдельный Railway-сервис рядом с m2ai.
