# P3 — Ontological GraphRAG + интерфейс общения · спецификация

> Встаёт на экспорт P2: graph/export/documents.jsonl + m2graph.json (/api/documents, /api/graph).

## Архитектура (Hydrate → Index → Retrieve → Answer)

1. Index. Из 245 entity-документов строится лексический индекс BM25 (чистый Python, без внешних
   зависимостей и без облачных эмбеддингов — детерминированно и бесплатно на Railway). Документ =
   title + определение + версии + имена связей.
2. Retrieve. По запросу: BM25 → top-k сущностей; затем graph-expansion (1-hop по рёбрам графа) —
   добавляем связанные сущности с меньшим весом. Это и есть «онтологическая» часть RAG: ответ
   собирается не из изолированных чанков, а из связного подграфа.
3. Answer. Сборка цитируемого ответа: определения топ-сущностей + их связи + источники-якоря
   (книга+страница, deep-link в PDF) + ссылки на статьи вики.

## Генерация (две дорожки)

- На сайте ключа к LLM нет → ответ экстрактивный (сборка из найденного, с цитатами). Это MVP.
- Полноценная генерация — через MCP: движок отдаётся как инструмент `m2_search`, агент (со своей
  моделью) делает генерацию по найденному подграфу. Это часть P0-E2 (KAN-258, MCP-сервер).

## Эндпоинты

- GET /api/rag?q=...&k=... — JSON: ранжированные сущности, расширение по графу, собранный контекст, цитаты.
- GET /ask?q=... — HTML-интерфейс «спросить»: ввод вопроса → ответ с определениями, связями, источниками, ссылками.

## Слайсы

- P3-S0 — движок BM25 + graph-expansion + /api/rag + /ask (MVP) — текущий.
- P3-S1 — эмбеддинги (опц.): гибрид BM25 + векторный ретрив (если подключим эмбеддинг-API/локальную модель).
- P3-S2 — MCP-инструмент m2_search в составе MCP-сервера (KAN-258), генерация на стороне агента.
- P3-S3 — оценка качества (faithfulness/citations), переранжирование.
